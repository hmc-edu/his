# 06 · 架构总览

> 杭州医学院 门诊 HIS 系统 · 实训文档

本文档帮你建立"鸟瞰视角"——理解整个系统由哪些部分组成、它们怎么对话、一次用户操作背后发生了什么。**改任何代码之前**都建议先看一遍。

---

## 1. 三层架构

整个系统是经典的"前后端分离 + 数据库"三层结构：

```
                 浏览器（Vue 3 + Element Plus）
                          │
                          │  HTTP / JSON / Authorization 请求头
                          ▼
                 后端（Spring Boot 3 + MyBatis）
                          │
                          │  JDBC SQL
                          ▼
                       MySQL 8
```

| 层 | 跑在哪 | 端口 | 职责 |
| --- | --- | --- | --- |
| 前端 | 浏览器 / 用户的电脑 | 5173（dev）/ 8000（prod nginx） | 界面渲染、表单校验、把用户操作变成 HTTP 请求 |
| 后端 | 学校服务器 / 你的电脑 | 8080 | 鉴权、业务逻辑、读写数据库、返回 JSON |
| 数据库 | 后端旁边 / 同一台机器 | 3306 | 持久化存储 |

> **为什么前后端分离？** 前端可以独立部署到 CDN，后端可以独立水平扩容；前端工程师与后端工程师并行开发；同一套后端 API 可以同时给 Web、小程序、App 用。

## 3. 后端分层（Java 这边）

后端使用经典的 **Controller / Service / Repository / Model** 四层：

```
┌──────────────────────────────────────────────────────────────┐
│  Controller —— HTTP 接口层                                    │
│  • 接受/返回 JSON                                              │
│  • 参数校验（@Valid）                                          │
│  • 角色校验（@SaCheckRole）                                    │
│  • 调用 Service，包成 R<T>                                     │
│  e.g. RegistrationController#create                          │
└─────────────────────────────┬────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────┐
│  Service —— 业务逻辑层                                        │
│  • 业务规则（状态机、金额计算、唯一性检查…）                   │
│  • 跨 Repository 协作                                          │
│  • @Transactional 事务边界                                     │
│  e.g. RegistrationServiceImpl#create                         │
└─────────────────────────────┬────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────┐
│  Repository —— 数据访问层（MyBatis Mapper）                   │
│  • 纯 SQL，无业务逻辑                                          │
│  • 注解写法（简单）或 XML 写法（复杂动态 SQL）                 │
│  e.g. RegistrationRepository#insert / selectById             │
└─────────────────────────────┬────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────┐
│  Model —— 数据库实体类（PO）                                  │
│  • 字段对应 SQL 表字段（驼峰 ↔ 下划线由 MyBatis 自动转）      │
│  e.g. Registration.java 对应 registration 表                 │
└──────────────────────────────────────────────────────────────┘
```

每一层的"形状"和"约束"都不同——要把对的逻辑放在对的层。详见 [07-后端开发指南](./07-后端开发指南.md)。

## 6. 鉴权流程：Sa-Token 在后端是怎么工作的

```
┌──────────┐                                  ┌─────────────────┐
│ 前端登录 │ ──POST /api/auth/login─────────▶ │ AuthController  │
└──────────┘                                  └────────┬────────┘
                                                       │
                                                       ▼
                                              ┌─────────────────┐
                                              │ AuthService     │
                                              │ • 查 sys_user   │
                                              │ • BCrypt 校验   │
                                              │ • StpUtil.login │
                                              └────────┬────────┘
                                                       │
                                                       ▼
       ┌──── tokenValue (UUID) ◀───────  Sa-Token 在内存/Redis 里记 token→userId
       │
       ▼
┌────────────┐ 之后每次请求带上：              ┌──────────────────┐
│  浏览器    │ ──Authorization: <token>─────▶ │ SaInterceptor    │
│ localStorage 存 token                       │ 拦 /api/**       │
└────────────┘                                │ • checkLogin     │
                                              │ • @SaCheckRole?  │
                                              └────────┬─────────┘
                                                       │ 通过
                                                       ▼
                                              ┌──────────────────┐
                                              │ Controller 方法   │
                                              └──────────────────┘
```

关键代码位置：

| 文件 | 作用 |
| --- | --- |
| `config/SaTokenConfig.java` | 注册拦截器、设置白名单（登录、Swagger） |
| `controller/AuthController.java` | 登录/登出/查当前用户 |
| `common/GlobalExceptionHandler.java` | 把 `NotLoginException` 翻译成 `R{code:401}` |
| 前端 `api/request.js` | 请求拦截器自动注入 token；401 自动跳登录 |

## 7. 关键设计点（必看）

### 7.1 统一响应包装

所有 Controller 返回值都包成 `R<T>`：

```json
{ "code": 0, "msg": "ok", "data": {...} }   // 成功
{ "code": 400, "msg": "用户不存在" }        // 业务异常
{ "code": 401, "msg": "未登录或登录已过期" }  // 鉴权失败
```

前端 `api/request.js` 拦截响应，**只在 `code==0` 时把 `data` 返回给调用方**，业务异常直接弹 toast。所以你在页面里写的：

```js
const reg = await createRegistration({ patientId, doctorId })
// reg 就是后端 data 里的 Registration，不需要你判断 code
```

### 7.2 状态机收口

`registration.status` 的所有切换只在 service 层完成，前端**永远不传 status 字段**。这避免了前端绕过业务规则直接改状态。

### 7.3 处方金额服务端计算

前端只传 `drugId / qty / days / dosage / frequency`，**不传金额**。后端 `VisitServiceImpl#prescribe` 拿 `drug.price` × `qty` 算 `subtotal`，再累加成 `total_amount`。这避免了前端被改后篡改价格。

### 7.4 MyBatis 双写法

- **注解**（`DepartmentRepository`）：简单 CRUD，SQL 短，写在接口方法的 `@Select` / `@Insert` 注解里
- **XML**（`PatientRepository` + `mappers/PatientMapper.xml`）：动态 SQL 多、SQL 长时用，便于格式化

两种你都要会读、会写。

## 8. 角色与权限

| 角色 | 可访问菜单 | 后端拦截方式 |
| --- | --- | --- |
| **ADMIN** | 全部 | `@SaCheckRole("ADMIN")` 标记 UserController |
| **RECEPTION** | 患者 / 挂号 / 挂号查询 | 接口级无角色限制（普通登录即可访问），由前端菜单决定可见性 |
| **DOCTOR** | 待诊队列 / 就诊页 | 同上 |

> **注意**：当前实现是"轻量 RBAC"，用一个 `role` 字段区分。要做完整的菜单权限/按钮权限，参见 [04-扩展练习](./04-扩展练习.md) 练习 5。

## 9. 常见问题（架构相关）

**Q：为什么不直接在 Controller 里写 SQL？**
A：那样 Controller 既要管 HTTP 又要管业务规则又要管 SQL，复杂度爆炸，无法测试。分层就是为了让每一层只负责一件事。

**Q：Service 接口和 Service 实现为什么要分开？**
A：方便 Mock 测试（写单元测试时可以注入假的 Service）、方便切换实现（比如未来切到分布式 / 缓存）。是 Spring 生态的强约定。

**Q：MyBatis 和 JPA 有啥区别？**
A：MyBatis 你自己写 SQL（控制力强、SQL 直观），JPA 自动生成 SQL（简单 CRUD 省事但复杂查询难写）。本项目用 MyBatis 是因为 HIS 系统经常要写复杂联表查询。

**Q：DTO 和 Model 有啥区别？**
A：Model（PO）字段和数据库表一一对应。DTO 是接口传输用的，前后端契约。比如 `LoginReq` 只有 `username/password`，不会有 `created_at` 等数据库字段。这样前后端契约清晰，也不会把不该返回的字段（比如密码）泄露出去。
